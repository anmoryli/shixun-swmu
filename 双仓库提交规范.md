# 双仓库提交规范

## 1. 远程仓库
本项目同时维护两个远程仓库，**每次提交都必须推送到两者**：

| 远程名 | 地址 | 说明 |
| --- | --- | --- |
| `origin` | `https://github.com/anmoryli/shixun-swmu.git` | GitHub 仓库 |
| `codearts` | `https://codehub.devcloud.cn-north-4.huaweicloud.com/9b1957ad04f844cfa48e66dd371a59bf/shixun-2.git` | 华为云 CodeArts 仓库（新建） |

> 新仓库（CodeArts `shixun-2`）从最初状态开始，已补齐全部历史提交，与 GitHub 仓库保持完全一致的历史。

## 2. 推送命令
每次提交后，必须依次推送到两个远程：

```bash
git push origin main
git push codearts main
```

- 若某个远程推送失败（如认证问题），需修复后单独补推，确保两者最终一致。
- 不得只推送其中一个仓库。

## 3. 提交粒度（最小化）
- **频繁提交**：写完一个最小功能单元就立即提交，不要积攒一大批改动后再提交。
- **粒度单位**：一次提交只做一件事，典型粒度包括
  - 新建一个函数 / 方法；
  - 实现一个功能点；
  - 修复一条代码扫描规则；
  - 补充一段文档。
- **禁止混入无关改动**：一个提交里只包含与本次目标直接相关的文件与代码行。

## 4. 提交信息格式
```
<类型>(<范围>): <简要描述完成的功能>
```

- **类型**：`feat`（新功能）/ `fix`（修复）/ `refactor`（重构）/ `style`（格式）/ `docs`（文档）/ `test`（测试）/ `chore`（构建/杂项）
- **范围**：受影响的模块，如 `backend`、`frontend`、`security`、`docs`。
- **描述**：用一句话说明本次提交**完成了什么功能**，而不是"修改了 xx 文件"。

示例：
```
fix(security): 登录令牌改为内存存储，不再写入 localStorage (G.WST.01)
refactor(backend): PageSupport 可空返回值改用 Optional (G.MET.06)
docs: 新增双仓库提交规范文档
```

## 5. 校验
每次推送前确保：
- 本地 `mvn -o clean compile`（后端）通过；
- 本地 `npm run build`（前端）通过；
- 涉及 Docker 的改动执行 `docker compose up -d --build` 同步更新镜像。

---
本规范自写入起长期有效，后续所有提交均须遵守。
