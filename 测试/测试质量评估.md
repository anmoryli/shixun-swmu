# 测试质量评估

| 文档项 | 内容 |
|---|---|
| 评估对象 | 慧医数字医疗应用系统当前仓库版本 |
| 评估方式 | 证据驱动的加权评分 + 强制发布门禁；总分不能覆盖关键门禁失败 |
| 证据截点 | 2026-07-13；只引用仓库中能够定位的报告、代码和配置 |
| 重要限制 | 本次评估不执行服务、Docker、浏览器或移动真机测试；“待验证”不等于“通过” |
| 编码约定 | UTF-8 中文 Markdown |

## 评分规则

| 评分 | 判定规则 | 证据要求 |
|---:|---|---|
| 5 | 该维度全部关键标准达到，且有当前版本自动化或可复核实测证据 | 报告含目标、时间、结果和版本标识 |
| 4 | 核心标准达到，但缺少容量、兼容性或纵深测试等非阻断证据 | 至少有完整核心链路报告 |
| 3 | 基础能力已验证，但证据过旧、覆盖不完整或仅适用于部分组件 | 可定位历史证据，并列出明确缺口 |
| 2 | 只有局部验证或主要依赖静态检查 | 有局部记录，无法支持整体结论 |
| 1 | 只有代码/配置可见，尚无执行证据 | 仅可证明“已实现/已设计” |
| 0 | 没有证据，或已知关键检查失败 | 必须阻断相关发布范围 |

| 总分区间 | 数值结论 | 使用条件 |
|---:|---|---|
| 90～100 | 建议发布 | 所有强制门禁均通过，且无未接受的 P0/P1 缺陷 |
| 80～89 | 有条件发布 | 强制门禁均通过；剩余风险有责任人、期限和回退方案 |
| 0～79 | 不建议完整发布 | 继续补测或修复；允许另行批准受限演示，但不得表述为全面验收通过 |

## 风险与改进优先级

| 优先级 | 风险 | 影响 | 建议闭环动作 | 完成判据 |
|---|---|---|---|---|
| P1 | 最新前端改动可能未进入正在运行的 Docker 镜像 | 用户仍看到旧页面，发布内容与源码不一致 | 每次前端变更后用新标签重建 Web 镜像，并记录提交号、镜像摘要、静态资源哈希 | 目标页面显示版本标识；容器摘要与发布记录一致；新旧切换可见 |
| P1 | 新旧 UI 缺浏览器端核心链路回归 | 样式覆盖可能遮挡按钮、表格、对话框或分页 | 在登录页、首页和八类管理页分别执行双模式冒烟 | 所有关键操作通过，P0/P1 为 0 |
| P1 | 390×844 移动端无 Android/iOS 实测 | 响应式代码可能在真实 Safari/Chrome 中溢出或触控困难 | 按 `移动应用测试.md` 执行两平台关键矩阵 | Android Chrome 和 iOS Safari 关键用例 100% 通过 |
| P2 | 只有单用户接口耗时，无容量结论 | 并发访问时可能超时或连接池耗尽 | 对登录、首页、药品分页做分级并发压测，单独使用测试环境 | 达到约定并发下错误率、P95 和资源阈值 |
| P2 | 在线拨测未落地 | 演示或生产异常不能及时发现 | 部署只读探针，接入指标存储、告警和恢复通知 | 连续运行至少 24 小时，有可查询曲线和一次告警演练 |
| P2 | 代码扫描结果只有 PDF，缺少结构化门禁 | 高危问题可能无法自动阻断发布 | 导出严重等级、规则、位置、处置状态和误报理由 | 高危未接受项为 0，持续集成可读取结果 |
| P3 | 前端缺自动化单元/E2E | 每次 UI 改动依赖人工重复检查 | 优先自动化登录、模式切换、首页和药品图片展示 | 自动化在两种 UI 模式下稳定通过，失败有截图/日志 |

## 评估结论

| 结论项 | 评估结果 |
|---|---|
| 暂定质量分 | 75/100；该分数来自明确评分规则，不代表当前运行环境已实测通过 |
| 后端/API 基线 | 现有证据充分支持功能回归、基础权限和单轮响应性能达到内部演示基线 |
| 完整发布判定 | **不建议按“完整版本已全面通过”对外发布**；G5、G6 未满足，G1～G4 还缺当前提交证据 |
| 可接受范围 | 如仅进行受控内部演示，可在负责人接受风险、确认 Docker 已含目标提交并完成桌面端冒烟后进行；不得把移动端标为已验收 |
| 转为“有条件发布”的最小条件 | 当前 Maven 8/8、API 57/57、前端构建成功、Compose 2/2 healthy、镜像版本可追溯、新旧 UI 核心链路和 390×844 双平台关键用例全部通过 |
| 转为“建议发布”的条件 | 在上述基础上关闭 P0/P1、完成代码扫描高危清零、至少 24 小时在线拨测，并形成可回退记录 |
