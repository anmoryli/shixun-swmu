## 2. 测试目标与质量原则

| 编号 | 测试目标 | 可检查的完成条件 | 优先级 |
|---|---|---|---|
| MB-01 | 证明管理员和医生可完成各自被授权的核心业务 | 需求追溯矩阵中的一级场景全部执行，管理员写操作和医生只读操作均有证据 | 最高 |
| MB-02 | 证明认证、会话与服务端权限不可由客户端状态绕过 | 无令牌、伪造令牌、过期令牌、退出后令牌、篡改本地角色和医生越权写入均被拒绝且数据不变 | 最高 |
| MB-03 | 证明八类业务数据及关联关系保持一致 | 页面、接口、数据库三方可核对；新增、修改、删除及引用冲突符合事务和约束设计 | 最高 |
| MB-04 | 证明新旧界面可以在全局稳定切换 | 登录页、首页和各管理页均可双向切换；刷新及重新登录后状态符合本地存储约定；切换不改变业务数据 | 高 |
| MB-05 | 证明新界面在常见桌面和移动视口中清晰、克制、可操作 | 无关键遮挡、横向溢出和不可读文本；键盘焦点、按钮语义、减少动画偏好有效 | 高 |
| MB-06 | 证明上传链路满足格式、大小、内容和持久化要求 | 仅真实 JPG/PNG、大小不超过 2 MB、尺寸不超过 8000×8000；随机文件名；重建容器后命名卷中的图片仍可访问 | 最高 |
| MB-07 | 证明构建产物和 Docker 页面来自同一提交 | 每次代码变更后重新构建镜像并部署；页面版本、静态资源及接口行为与目标提交一致，不复用陈旧前端产物 | 最高 |
| MB-08 | 证明主要操作满足响应目标 | 主要操作目标 1 秒内，单次硬上限 3 秒；统计口径、样本量和环境随报告记录 | 高 |

## 9. 缺陷分级与闭环

| 等级 | 判定示例 | 发布规则 | 复测要求 |
|---|---|---|---|
| 致命 | 无法登录、数据批量破坏、权限完全失守、凭据泄漏、服务整体不可用、无法回滚 | 必须暂停相关测试和发布，未关闭数量必须为 0 | 修复后执行根因场景、全量最高优先级回归和数据核对 |
| 严重 | 管理员核心增删改失败、医生可写入、事务不一致、上传绕过安全限制、Docker 持续展示旧版本 | 阻断发布，未关闭数量必须为 0 | 定向复测、关联模块回归、部署重建验证 |
| 一般 | 非核心校验错误、局部显示/兼容问题、有明确绕行方式 | 原则上发布前关闭；遗留须书面评估 | 修复场景及同类边界用例复测 |
| 轻微 | 文案、间距、非关键视觉差异，不影响操作和数据 | 可经评审后遗留到下一迭代 | 核对修改页面及两套主题，避免经典界面回归 |
| 闭环状态 | 新建 → 已确认 → 修复中 → 待复测 → 已关闭；无法修复或延期必须记录原因、影响和批准人 | 任何关闭均需要可重复步骤、实际结果和证据 | 代码修复后必须重建 Docker，不接受仅开发服务器验证 |

## 10. 暂停与恢复规则

| 触发类型 | 暂停条件 | 暂停范围 | 恢复条件 |
|---|---|---|---|
| 构建阻塞 | 前端或后端无法从目标提交构建，或 Docker 镜像不是最新提交 | 暂停全部依赖该产物的动态测试 | 构建修复、镜像重建、提交对应关系核对、基础烟测通过 |
| 环境阻塞 | MySQL、Redis、后端健康检查持续不可用，或测试账号失效 | 暂停接口、页面和数据测试 | 环境恢复，健康、登录、权限和只读查询烟测通过 |
| 数据风险 | 发现测试写入生产数据、临时数据无法定位、异常级联删除或共享数据被污染 | 立即暂停所有写测试 | 完成备份核对和影响评估，隔离数据恢复，清理方案获批准 |
| 安全风险 | 医生越权写入、无令牌访问受保护接口、上传任意文件或报告暴露密码/完整令牌 | 暂停发布相关测试并升级为致命缺陷 | 漏洞修复，负向用例及全权限矩阵通过，证据完成脱敏 |
| 共因失败 | 大量用例因同一基础故障失败，继续执行只会产生无效结果 | 暂停受影响测试集 | 根因修复后先跑最小烟测，再从首个受影响用例恢复 |
| 恢复记录 | 任何暂停均记录时间、基线、已执行范围、未清理数据和负责人 | 不适用 | 恢复时生成新测试轮次，不覆盖暂停前原始结果 |

## 11. 测试数据与清理计划

| 数据类别 | 准备方式 | 隔离与安全要求 | 清理方式 |
|---|---|---|---|
| 管理员账号 | 使用专用测试管理员，密码由本机私有环境提供 | 不写入仓库、报告、截图或命令历史；仅授予测试所需权限 | 不删除基线账号；测试后退出并确认令牌失效 |
| 医生账号 | 通过管理员流程创建带唯一运行标识的临时医生 | 覆盖初始密码、重置密码、旧密码和旧令牌失效 | 先删除其依赖数据，再删除医生和账号，核对审计策略 |
| 业务主数据 | 名称加入唯一运行标识，使用合法公司、城市、销售地点关联 | 不修改原始演示记录；并发测试不得复用同一标识 | 按药品/政策 → 医生/销售地点 → 公司/城市的依赖逆序清理 |
| 边界与非法数据 | 空值、超长、特殊字符、无效编号、重复手机号、越界坐标、伪造令牌 | 仅通过公开接口输入，不直接破坏共享数据库约束 | 失败请求后核对数据库无新增；成功的边界记录纳入正常清理 |
| 图片数据 | 准备小于、等于和大于 2 MB 的真实/伪造 JPG、PNG；准备 8000 和 8001 像素边界样本 | 不含个人或医疗敏感信息；记录文件哈希；文件名不携带路径 | 业务记录经接口删除；无文件删除接口时仅在隔离上传卷按批准清单清理 |
| 中文编码数据 | 使用中文名称、地址、政策内容及中英文混合文本 | 请求、响应、数据库和 Markdown 报告统一采用 UTF-8 | 清理后抽查无乱码残留；报告以 UTF-8 保存 |

## 12. 质量指标与发布阈值

| 指标 | 计算口径 | 计划阈值 | 数据来源 |
|---|---|---|---|
| 需求覆盖率 | 已有测试设计和用例的需求数 ÷ 范围内需求总数 | 100% | 需求追溯矩阵 |
| 用例执行率 | 已执行用例数 ÷ 计划用例数 | 发布前 100% | 测试执行记录 |
| 总体通过率 | 通过用例数 ÷ 已执行用例数 | 不低于 95%，且最高/高优先级必须全部通过 | 测试执行记录 |
| 缺陷关闭率 | 已关闭缺陷数 ÷ 已确认缺陷数 | 致命和严重 100%；其他按发布评审结论 | 缺陷清单 |
| 权限负向通过率 | 被正确拒绝且数据不变的越权用例数 ÷ 权限负向用例数 | 100% | 接口报告、数据库核对 |
| 性能达标率 | 响应不超过 1 秒的主要操作数 ÷ 主要操作样本数 | 目标 100%；任何单次不得超过 3 秒 | 带时间戳的接口/页面性能记录 |
| 部署一致率 | 已完成目标提交镜像重建和烟测的变更轮次 ÷ 总变更轮次 | 100% | Docker 构建与部署记录 |
| 数据清理率 | 已确认清理的临时业务记录数 ÷ 已创建临时业务记录数 | 100%；隔离上传卷例外单列 | 运行标识查询和清理记录 |

## 13. 风险与应对

| 风险 | 可能影响 | 概率/影响 | 预防与应对 |
|---|---|---|---|
| Docker 缓存或旧静态产物未更新 | 代码已改但 9092 仍显示旧页面，测试结论针对错误版本 | 高/高 | 每次变更后使用 `docker compose up -d --build`；清缓存复核关键静态资源；在执行记录中同时写提交和镜像构建时间 |
| 新主题全局 CSS 级联覆盖经典界面 | 表格、弹窗、按钮或登录页出现不可读、遮挡和功能回归 | 中/高 | 每个核心页面在两套主题各执行；保留经典界面基线截图；按公共组件优先修复 |
| 共享远程 MySQL/Redis 被并发测试污染 | 用例相互干扰、误删数据、结果不可重复 | 中/高 | 使用唯一运行标识、禁止同环境并发整套回归、逆序清理、执行前备份和基线计数 |
| 上传接口没有文件删除能力 | 测试图片在命名卷累积 | 高/中 | 仅使用隔离卷；记录文件名和哈希；由部署负责人按批准清单清理，不直接操作业务环境 |
| 角色只在前端隐藏但服务端未拒绝 | 医生可通过直接请求修改数据 | 低/极高 | 权限判定以服务端响应和数据不变为准；所有写接口逐一做医生越权测试 |
| 浏览器本地存储被禁用或损坏 | 主题状态不能持久化或页面报错 | 中/中 | 设计存储异常降级用例；当前会话仍应可切换，非法值应回到默认新界面 |
| 外部地图密钥和目标 Linux 凭据缺失 | 地图底图及目标主机部署无法验证 | 高/中 | 明确列为范围外/环境限制，不用模拟结果替代；验证已有坐标和交付配置 |
| 报告包含密码、令牌或业务数据 | 安全泄露，报告不可分发 | 低/极高 | 报告不记录请求头、密码和完整令牌；截图脱敏；发布前二次检查 |
| 中文路径或编码处理不一致 | 文件名、内容、日志或报告乱码 | 中/高 | 仓库文档统一 UTF-8；PowerShell 读取显式使用 UTF-8；提交前检查中文文件名和替换字符 |

## 15. 变更与批准规则

| 事项 | 规则 |
|---|---|
| 基线变更 | 测试开始后任何代码、配置、数据库迁移或依赖变化都要形成新测试轮次，登记新提交及影响范围，不覆盖旧结果 |
| 用例变更 | 增删最高/高优先级用例必须经测试负责人和需求/验收人员确认，并同步追溯矩阵 |
| 例外发布 | 未达到准出标准时不得默认为通过；如确需发布，必须记录遗留缺陷、用户影响、临时措施、回滚触发条件和批准人 |
| 结果真实性 | “未执行”“阻塞”“跳过”必须按原状态记录，不得填写为通过；既有历史报告不得代替本轮实际执行 |
| Docker 同步 | 每次代码修改完成后，先提交到目标分支，再重建并更新 Docker 环境；测试结果必须来自更新后的容器 |
